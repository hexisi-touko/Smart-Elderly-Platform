# 智慧养老平台：个性化健康数据校验算法设计 (v1.0)

> **面向对象**: AI 编程助手 (Agent)
> **关联业务**: 高并发设备数据上报、个性化阈值预警、慢病基线修正

## 1. 算法背景与目标
针对老年人个体差异大、慢性病背景复杂的特点，废弃统一的“静态阈值”方案。采用“三级优先级 + 动态基线波动”算法，旨在解决：
1. **防误报**: 允许慢病老人（如高血压）在安全范围内有更高的常态血压。
2. **防漏报**: 即使指标在安全范围内，若发生相对于自身基线的“剧烈突变”，必须报警。

## 2. 数据结构设计 (Redis Hash)
为支撑 1000 QPS 并发，校验逻辑必须在 Redis 内存中完成。
**Key**: `elderly:threshold:{elderlyId}`
**Fields**:
- `sysBpMax / sysBpMin`: 综合判定后的收缩压上下限。
- `diaBpMax / diaBpMin`: 综合判定后的舒张压上下限。
- `avgSysBp`: 过去 7 天收缩压的移动平均值（动态基线）。
- `chronicTags`: 老人关联的慢病标签（如：高血压、糖尿病）。

## 3. 校验逻辑流转 (Logic Pipeline)

### 第一层：三级优先级绝对值校验
上报数据需依次匹配以下优先级获取判定标准：
1. **[P1 个性化设定]**: 若 `t_elderly_health_threshold` 有记录，优先使用。
2. **[P2 慢病修正]**: 若无 P1 且存在“高血压”标签，收缩压上限自动从 140 宽放至 155。
3. **[P3 国家标准]**: 默认兜底（如血压 90~140, 血糖 3.9~6.1）。

### 第二层：动态波动率校验 (防止指标突降)
即使数据通过第一层校验，仍需执行此公式：
- **偏离度公式**: $Deviation = \frac{|CurrentValue - avgSysBp|}{avgSysBp}$
- **判定阈值**: 若 $Deviation > 0.2$ (即偏离平稳基线 20%)，判定为“数据异常突变”。

## 4. 状态标记规则
- **正常 (Normal)**: 通过第一层校验 AND 偏离度 ≤ 0.2。
- **严重异常 (Critical)**: 未通过第一层校验 OR 偏离度 > 0.2。
  - *动作*: 立即触发 WebSocket 推送并在 `t_health_record` 标记 `data_status = 2`。

## 5. 后端实现约束 (针对 Service 层)
1. **禁止实时查库**: 校验过程仅允许操作 Redis。
2. **异步入库**: 校验后将数据丢入 `Redis List` 队列，由定时任务批量同步至 MySQL。
3. **解密逻辑**: 从缓存读取数据前，需确保 `elderlyId` 匹配，并预留解密接口调用位置。
